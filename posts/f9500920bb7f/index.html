匹配含有脱敏符号的字符串

发表于 2024-01-23 分类于 笔记 ， 前端

最近遇到一个需求，简要描述就是：有一组脱敏后的数据需要展示为选项，同时要求前端能够通过输入筛选出部分匹配的项

特定格式版本

最初的想法是根据需求指定长度的格式去做处理，省的去判断 * 符号位置。但是由于 mock 造出的数据比较奇葩，各种长度的字符串都有；同时也考虑到以后要是数据格式变化，还得特地去调整代码，于是便写了稍微通用一点的处理方法。

可支持任意数量的脱敏符号和明文字符，限制为仅包含一串连续的脱敏字符
例如：A****B，****A，A***

function isMatched(input, option) {
    try {
        for (let i = 0; i <= option.length - input.length; i += 1) {
          // 待匹配的部分字符串
          const toMatchStr = option.slice(i, input.length + i);
          // 寻找 * 符号
          const starStartIdx = toMatchStr.indexOf('*');
          const starEndIdx = toMatchStr.lastIndexOf('*');
          // 待匹配的部分全为 * 符号，跳过匹配
          if (starStartIdx === 0 && starEndIdx === toMatchStr.length - 1) {
            continue;
          }
          // 待匹配的部分不全为 * 符号，替换输入内容对应位置的字符为 * 后再进行匹配
          let formattedInput = input;
          if (starStartIdx !== -1) {
            const starCount = 1 + starEndIdx - starStartIdx;
            formattedInput = input.slice(0, starStartIdx) + '*'.repeat(starCount) + input.slice(starEndIdx + 1);
          }
          // 匹配成功
          if (formattedInput === toMatchStr) return true;
        }
    } catch (e) {
        console.error('匹配异常', e);
    }
    return false;
}

时间复杂度 O(n)，空间复杂度 O(1)，看起来还不错

通用版本

上述仅能支持一串脱敏符号的场景，若有多串脱敏符号，常规做法可以一个个字符匹配
例如：***A***B***

function isMatched(input, option) {
    try {
        for (let i = 0; i <= option.length - input.length; i += 1) {
          // 待匹配的部分字符串
          const toMatchStr = option.slice(i, input.length + i);
          // 待匹配的部分全为 * 符号，跳过匹配
          if (toMatchStr === '*'.repeat(toMatchStr.length)) {
            continue;
          }
          // 待匹配的部分不全为 * 符号，逐个判断字符
          let matched = true
          for (let j = 0; j <= toMatchStr.length; j += 1) {
              if (toMatchStr[j] !== '*' && toMatchStr[j] !== input[j]) {
                matched = false
                break
              }
          }
          // 匹配成功
          if (matched) return true;
        }
        return false;
    } catch (e) {
        console.error('匹配异常', e);
    }
}

时间复杂度 O(n^2)，空间复杂度 O(n^2)，好像勉强也还行。当然还有可以优化的地方，比如判断待匹配部分有没有 * 号，没有则可以直接全等匹配，尽量避免 for 循环嵌套。

更通用的动态规划版本？

具体思路可以参考拓展的第一篇文章，不同的是，此处的字符串只要匹配上一部分就可以。但是实际操作发现，部分匹配和 dp 思想有冲突，仍需要先将待匹配字符串（pattern）做分割再用 dp 计算，不过这样的话复杂度都会变成 O(n^3)，感觉有些杀鸡用牛刀。
